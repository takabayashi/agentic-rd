Wikipedia Edit-Triage Agent

A locally-runnable system that ingests the Wikipedia recent-changes firehose, triages each edit with a multi-step LLM reasoning loop (Redpanda Connect + Ollama), and serves the results on a live dashboard.

3 Ingest flow — Wikipedia SSE planned

Phase 4–5: pull the SSE firehose, strip the data: prefix, fail-closed on heartbeats, then filter and project to a clean schema.

sequenceDiagram
  participant W as Wikipedia SSE
  participant C as Connect http_client
  participant B as Bloblang transform
  participant K as Cache (dedupe)
  participant N as Next stage (LLM)

  C->>W: GET stream (User-Agent required)
  W-->>C: text/event-stream frames
  loop each line (lines scanner)
    C->>B: raw frame
    B->>B: strip "data:" prefix
    B->>B: parse_json().catch(deleted())
    alt heartbeat / non-JSON
      B-->>C: drop (fail-closed)
    else valid edit
      B->>B: keep type=edit, bot=false, namespace=0
      B->>B: project schema + size_delta
      B->>B: epoch to ISO event_ts
      B->>K: check rev_id
      alt seen
        K-->>B: drop duplicate
      else new
        K-->>B: add rev_id
        B->>N: clean edit record
      end
    end
  end

4 Multi-step LLM reasoning loop planned

Phase 7–8: a cheap first-pass classification, then a confidence-gated escalation pass that pulls richer context. Output is parsed defensively and normalized to a fixed enum.

sequenceDiagram
  participant E as Clean edit
  participant B1 as branch (pass 1)
  participant O as Ollama (ollama_chat)
  participant P as Robust parse + normalize
  participant S as switch (confidence)
  participant B2 as branch (pass 2, escalate)
  participant Sink as Sink (topic + UPSERT)

  E->>B1: request_map builds prompt (title, comment, size_delta)
  B1->>O: chat completion
  O-->>B1: raw model text
  B1->>P: extract first {...}, fallback {}
  P->>P: normalize label to enum, default unclear
  P->>P: coerce confidence to number
  P-->>S: {label, confidence}
  alt confidence < CONFIDENCE_THRESHOLD or label == unclear
    S->>B2: escalate with richer diff context
    B2->>O: chat completion
    O-->>B2: raw model text
    B2->>P: reuse robust parse
    P-->>Sink: {label, confidence, escalated=true}
  else high confidence
    S-->>Sink: {label, confidence, escalated=false}
  end

5 Connect pipeline stages planned

Phase 4–9: how a single edit flows through the Redpanda Connect config — input, processors (branches), and the dual sink.

flowchart TD
  in["input: http_client (SSE, stream.enabled, lines)"]
  d1["processor: strip data: + parse_json.catch(deleted)"]
  d2["processor: filter type/bot/namespace + project"]
  d3["processor: cache dedupe on rev_id"]
  br1["branch: pass-1 classify (ollama_chat)"]
  pr1["result_map: graft {label, confidence}"]
  sw{"switch: confidence < threshold OR unclear"}
  br2["branch: pass-2 escalate (ollama_chat)"]
  pr2["result_map: graft + escalated=true"]
  out1["output: kafka topic wiki.edits.classified (key=rev_id)"]
  out2["output: sql_insert UPSERT classified_edits"]

  in --> d1 --> d2 --> d3 --> br1 --> pr1 --> sw
  sw -- escalate --> br2 --> pr2 --> out1
  sw -- keep --> out1
  pr2 --> out2
  out1 --> out2

7 Domain model

The edit view-model rendered by the dashboard and the classified_edits table (Phase 2–3). rev_id is the primary key powering UPSERT-based correction.

classDiagram
  class ClassifiedEdit {
    +int rev_id PK
    +string title
    +string editor
    +string comment
    +Label label
    +float confidence
    +bool escalated
    +int size_delta
    +string uri
    +timestamptz event_ts
    +timestamptz classified_at
  }
  class Label {
    <<enumeration>>
    vandalism
    substantive
    trivia
    unclear
  }
  ClassifiedEdit --> Label : CHECK constraint
  note for ClassifiedEdit "rev_id PK enables ON CONFLICT DO UPDATE;\ntitle + comment are untrusted (autoescaped in UI)"

8 Service lifecycle & startup ordering

Today only webapp runs under Compose with a healthcheck. Planned services gate on each other via healthchecks so docker compose up is robust to cold starts.

stateDiagram-v2
